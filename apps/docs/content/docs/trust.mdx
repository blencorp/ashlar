Ashlar's trust model starts with the source installed in your app. The CLI writes
a lockfile, records hashes, and exposes verification commands so changes are
visible.

## Verify installed source

<AshlarCommand args="verify" />

`verify` checks:

- the installed capsule version in `ashlar-lock.json`;
- the registry capsule hash;
- the copied files in your project;
- local edits that no longer match the original source hash.

## Audit before release

<AshlarCommand args={'audit --policy all "src/**/*.{html,tsx,jsx,vue,svelte}"'} />

## What you can claim

- You are using source-owned Ashlar starting points.
- Installed files can be checked against registry metadata.
- Theme adapters cite public design-system sources.
- The current foundation capsules are v0.0 prototype material with explicit
  evidence boundaries.

## What you cannot claim

- Ashlar replaces an agency design system.
- Ashlar is endorsed by a government agency.
- A component is accessibility-stable without completed evidence records.
- A theme is accurate because it looks close in a screenshot.

## Release gates

The public release path is intentionally stricter than a local build:

- public npm packages must exist and install through npm, pnpm, and bun;
- npm provenance must verify for the exact published package versions;
- capsule trust material must be published and verifiable by consumers;
- external review records must match the evidence being claimed.

Until those pass, Ashlar can ship only as an honest public prototype.

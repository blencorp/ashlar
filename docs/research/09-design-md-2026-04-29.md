# DESIGN.md research note

Date: 2026-04-29

## Summary

`DESIGN.md` is worth tracking as an emerging AI-design handoff convention. It is not a W3C standard, but it is no longer just an informal filename: Google Labs has a public `google-labs-code/design.md` repository, Apache-2.0 licensing, a package, a linter/diff/export CLI, examples, and a 0.1.0 release. The repo describes DESIGN.md as a format for giving coding agents a persistent, structured understanding of a design system.

Ashlar should adopt DESIGN.md as an **agent-facing export/toolbelt artifact**, not as the canonical design-token source.

## What DESIGN.md is

The Google Labs format combines:

- YAML front matter for machine-readable tokens: colors, typography, radius, spacing, component tokens.
- Markdown prose for rationale: overview, colors, typography, layout, elevation, shapes, components, do's and don'ts.
- A CLI surface: `lint`, `diff`, `export`, and `spec`.
- Export targets including Tailwind and DTCG.

That makes it complementary to Ashlar's current AI-native stack:

| Artifact | Role |
|---|---|
| DTCG tokens | Canonical token source for Ashlar and agency themes. |
| Extended CEM | Canonical component contract, accessibility requirements, anti-patterns, examples. |
| Evidence packets | Canonical accessibility/security/provenance claims. |
| AGENTS.md | Repo-local behavior instructions for coding agents. |
| DESIGN.md | Repo-local visual/design intent for coding agents. |
| MCP | Query and validation interface over the canonical data. |

## Maturity assessment

Strong signals:

- Public Google Labs repository and package.
- Apache-2.0 license.
- Active ecosystem forming around DESIGN.md libraries, generators, and MCP-style usage.
- The format matches the problem Ashlar already cares about: preventing agents from inventing visual rules.

Weak signals:

- Format is explicitly alpha.
- It is not a standards-body specification.
- Current schema is simpler than DTCG and Ashlar's planned token/component/evidence model.
- Community libraries often document "inspired by" brand systems, which can create licensing/trademark and design-integrity risks.

Conclusion: use it as an interoperability/export layer, not as a source of truth.

## Recommendation for Ashlar

Add DESIGN.md generation to `ashlar init`.

Default behavior:

```bash
ashlar init
```

Creates:

```text
ashlar.config.json
ashlar-lock.json
AGENTS.md
DESIGN.md
```

`DESIGN.md` should be generated from:

- Installed or default Ashlar DTCG tokens.
- Component CEM summaries.
- Policy-pack design constraints.
- Accessibility do/don't rules.
- Government UI/UX principles: plain language, restrained density, trustworthy civic tone, visible focus, forced-colors support, no decorative novelty.

The generated file should say:

```markdown
<!-- Generated by Ashlar. Canonical values come from DTCG tokens, CEM, and evidence packets. Edit agency theme tokens first; regenerate DESIGN.md with ashlar design sync. -->
```

Add commands in v0.0:

```bash
ashlar design sync        # regenerate DESIGN.md from tokens/CEM/policy packs
ashlar design lint        # run @google/design.md lint if available, plus Ashlar-specific checks
ashlar design export dtcg # optional bridge from DESIGN.md back to token experiments, not canonical publish path
```

## Why this improves DX

- Agents get visual guidance without needing to query MCP first.
- `DESIGN.md` gives Claude/Cursor/Copilot/Codex a familiar root-file hook.
- Designers and developers can review one readable file for tone, density, layout, component styling, and anti-patterns.
- It reduces generic AI UI output while preserving Ashlar's real source of truth in tokens/CEM/evidence.

## Guardrails

- Never let DESIGN.md override accessibility evidence or policy-pack rules.
- Do not ingest arbitrary public DESIGN.md files into federal projects without provenance review.
- Treat DESIGN.md imports as drafts: convert to DTCG tokens, validate contrast/forced-colors, then regenerate a clean Ashlar DESIGN.md.
- Avoid brand-clone libraries for public-sector work unless licensing and design ownership are clear.

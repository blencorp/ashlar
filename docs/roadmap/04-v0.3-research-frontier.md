# v0.3+ — Research frontier (ongoing)

The research-and-validation phase, not a fixed calendar track. Each frontier item is independently scoped and ships only when it passes a "validated demand + prototyped + measurable improvement" gate.

This phase exists to keep Ashlar architecturally a generation ahead, while protecting the stable v0.2 foundation from premature inclusion.

## Goal in one sentence

Validate which architectural primitives from the frontier (signals at platform level, resumability, effect systems, event-sourced patterns, CRDT-friendly patterns) deliver real improvement in real consumer projects, and integrate them as they prove out.

## Tracks

### Track 1 — Signals at platform level

**Status**: Ready to integrate when TC39 Signals reaches Stage 3 and ships.

**Work**:
- Replace internal Solid- / Preact-style signals with platform `Signal` and `Computed`.
- Update Lit shells and machines to consume platform signals.
- Migration codemods for any consumer code that touched signal APIs.

**Trigger**: TC39 Signals proposal at Stage 3, with at least one shipped browser implementation.

### Track 2 — Effect-system-typed accessibility

**Status**: Research / prototype.

**Work**:
- Prototype an Effect-TS-style API for declaring component a11y requirements.
- Build a worked example: Dialog component declaring `requires(FocusTrap)` and `requires(InertSiblings)`.
- Usability test with real product engineers.
- Document tradeoffs.

**Decision criteria**:
- Does it make consumer code clearer or muddier?
- Does it catch real bugs in real codebases?
- Does it integrate with TypeScript's existing inference acceptably?

**If yes**: integrate as opt-in for interactive components in v0.4 or later. **If no**: document the lesson and move on.

### Track 3 — Resumability-friendly render mode

**Status**: Architecture-compatible (interactive components serialize state to data attributes); runtime exploitation pending.

**Work**:
- Prototype an `@blen/ashlar-resume` package that exploits the data-attribute serialization for client takeover without re-execution.
- Validate with at least one Qwik-style or framework-neutral resumability runtime.
- Measure: time-to-interactive on slow devices; comparison with hydration baseline.

**Trigger**: a resumability runtime reaches enterprise-grade production maturity (Qwik, post-Qwik successor, or framework-neutral standard).

### Track 4 — Event-sourced patterns expansion

**Status**: One pattern (application-review) ships event-sourced in v0.2. Expansion to other patterns is the v0.3+ track.

**Work**:
- Document the event-sourcing primitives that v0.2's pattern depends on.
- Build community RFC process for proposing event-sourced versions of other service patterns.
- Ship event-sourced versions of: eligibility-check, document-upload, account-creation, with codemods from non-event-sourced predecessors.

**Decision criteria**:
- Real consumer demand (audit-heavy gov flows asking for it).
- Performance and complexity tradeoffs measured.

### Track 5 — CRDT-friendly collaborative patterns

**Status**: Pattern category, not a runtime requirement.

**Work**:
- Build patterns for: collaborative case review, simultaneous form editing, conflict-resolution UX, optimistic-update visuals.
- Each pattern is CRDT-engine-agnostic — works with Yjs, Automerge, ElectricSQL, Replicache, or PowerSync.
- Document integration guides per CRDT engine.

**Trigger**: real consumer (likely a state digital service or federal field-services team) with a real CRDT integration ready to design-partner.

### Track 6 — Independent v1.0 audit

**Status**: Required for v1.0.

**Work**:
- Independent third-party security audit of the registry, signing, and CLI.
- Independent third-party accessibility audit of all stable components and patterns.
- Independent third-party performance audit (bundle, runtime, accessibility-tree complexity).
- Public report.

**Trigger**: v1.0 readiness review.

### Track 7 — Native mobile (out indefinitely unless demand emerges)

**Status**: Not on the roadmap.

**Conditions for reactivation**:
- A real federal field-services or kiosk consumer with a hybrid native-web stack asks.
- The cost of supporting React Native or Lynx adapters is justified by clear adoption.
- We have engineers who genuinely understand the native a11y landscape (TalkBack semantics, VoiceOver semantics, native focus management).

**Until then**: web-first. PWA-friendly markup primitives and service patterns address the "make it feel native enough" needs.

## What ships when

There is no fixed v0.3 / v0.4 / v0.5 calendar sequence. Tracks land independently as their gates open. Each landing is its own minor or major release.

The Ashlar core team commits to:

- Gate-driven review of all tracks: which ones have gates opening, which need more research, which should be cut.
- Public RFC process for any track promotion to "shipping in next release."
- No track work is allowed to destabilize v0.2.

## Decision framework

For every frontier item, the same five questions apply:

1. **Validated demand?** Real consumer ask, or strategic differentiation, or both?
2. **Prototyped?** Smallest thing that proves the bet on a single component or pattern.
3. **Measurable improvement?** Better DX, better a11y, better performance, better auditability — with numbers.
4. **Documented decision?** ADR explaining why we adopted (or rejected) at the gate.
5. **Major version?** Architectural shifts go into a major version with codemods.

If any of the five is missing, the track waits.

## What this phase is not

- **Not a wish list.** Items are assessed against engineering and product cost. Saying no is healthy.
- **Not exhaustive.** Other primitives (logic-based form validation, aspect-oriented accessibility, TLA+ for invariants) are mentioned in [`research/06-statecharts-and-frontier.md`](../research/06-statecharts-and-frontier.md). They are not actively tracked here. They are worth knowing about; they are not commitments.
- **Not the path to v1.0.** The path to v1.0 is v0.0 → v0.1 → v0.2 plus track 6 (independent audit). Tracks 1–5 are post-1.0 unless they prove out earlier.

## How tracks change Ashlar

If a track lands, it changes Ashlar in a specific direction:

- **Signals at platform level**: Ashlar becomes future-proof against framework consolidation.
- **Effect-system a11y**: Ashlar catches real accessibility bugs at compile time.
- **Resumability**: Ashlar ships measurably faster government UIs on slow devices.
- **Event-sourced patterns**: Ashlar becomes uniquely useful for audit-heavy federal flows.
- **CRDT-friendly patterns**: Ashlar serves collaborative gov workflows that no current DS does.

Each is a category-shift from "open-source design system" toward "category-defining UI architecture for the next decade."

## What v1.0 looks like

When Ashlar reaches v1.0:

- 25–30 stable components.
- 6+ service patterns.
- application-block templates for Nunjucks, Twig, Jinja, ERB, plain HTML.
- framework adapters for React, Vue, Svelte, Solid.
- Independent third-party security and accessibility audits passed.
- 3+ contributing organizations.
- 100+ teams running Ashlar in production.
- Track 1 (signals) integrated.
- Tracks 2–5 in various stages of validation; some may have landed.

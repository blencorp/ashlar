# v0.0 — Foundation

The smallest viable product that proves the architecture. Internal alpha plus two or three external design partners. **No public release.**

The first code slice already exists: workspace, CLI skeleton, capsule schema, Button capsule, plain HTML example, and CI. The next slice must turn that prototype into a standards-and-evidence foundation before adding many components.

If v0.0 cannot meet its success criteria, the architecture needs revision before further investment.

## Goal in one sentence

Demonstrate that a developer can go from zero to a working, accessible federal-service form through a short deterministic CLI path, with content-addressed capsules, standards-pack validation, evidence packets, a working drift-management protocol, and AI-readable component manifests — and that the resulting bundle is materially smaller than shadcn equivalents.

## Deliverables

### Tokens and CSS foundation

- DTCG 2025.10 source for primitive, semantic, and component tokens.
- Terrazzo compiler integration emitting CSS variables, Tailwind v4 `@theme`, typed TypeScript.
- One default theme using `oklch()` and `light-dark()`.
- Cascade-layer architecture (`@layer ashlar.reset, ashlar.tokens, ashlar.base, ashlar.components, ashlar.utilities, ashlar.overrides`).
- Platform feature tier policy for `dialog`, `popover`, `light-dark()`, `@scope`, anchor positioning, forced colors, and fallback requirements.

### Federal standards pack

First policy-pack prototype:

- HTML page title rule.
- Meta description rule.
- Federal government banner placement rule.
- Identifier / required-link completeness rule.
- Accessibility statement and privacy policy link checks.
- SARIF output plus artifact fallback for repositories without GitHub Code Security enabled.

### L0 capsules (CSS+HTML, zero JS)

Thirteen components, all platform-driven:

1. Button
2. Link
3. Alert
4. Banner (gov shell)
5. Identifier (required-link shell)
6. Card
7. FormField (Label + Input wrapper + ErrorMessage)
8. TextInput
9. Textarea
10. Checkbox
11. Radio
12. Dialog (using `<dialog>`)
13. Details (using `<details name>`)

Each with: CSS, HTML reference, Nunjucks template, extended CEM, basic evidence packet, platform-feature metadata, ast-grep rules from CEM, and `*.docs.md`.

### L1 capsule

One Lit custom element, to prove the WC pipeline:

- ComboBox (using a Zag machine)

With: machine, Lit shell, React adapter (auto-generated from CEM), full extended CEM, accessibility evidence, ast-grep rules.

### Registry and CLI

- Static JSON registry served from a Vercel/Cloudflare CDN endpoint.
- Sigstore-signed capsules.
- CLI (Node + ESM) with: `init`, `search`, `view`, `add`, `diff`, `update`, `audit`, `verify`, `evidence`.
- `ashlar-lock.json` working end-to-end.
- Three-way merge protocol functional with at least 30 instrumented test updates.

### Evidence integration

- Evidence schema maps WCAG criteria, manual tests, automated tests, and Section 508 ICT Baseline test references where applicable.
- `ashlar evidence <component>` explains what has been tested and what remains untested.
- `ashlar audit --explain` links findings to component evidence and policy-standard sources.

### AI integration (minimal)

- Extended CEM for all 13 capsules.
- `AGENTS.md` template scaffolded by `ashlar init`.
- `DESIGN.md` template scaffolded by `ashlar init`, generated from Ashlar tokens/CEM/policy packs as an agent-facing export.
- `ashlar design sync` regenerates DESIGN.md after token or component changes.
- llms.txt at the docs site root.
- **No MCP server yet.** Defer to v0.1.

### Demo apps

Three real applications built on the same capsules:

1. Next.js form-flow demo (eligibility-style form with page metadata and standards-pack rules).
2. Plain HTML reference page (banner + identifier + card + form).
3. Drupal Twig integration test (one Ashlar pattern rendered as Twig partial).

All three render the same components with the same DOM contract and the same accessibility evidence.

### Docs

- `docs/` package fully populated (already drafted; refines as code lands).
- Static docs site (Astro, server-rendered, llms.txt at root).
- Per-component documentation pages with extended CEM as machine-summary block.

## Success criteria

A v0.0 release ships only if **all** of the following are true:

- [ ] `ashlar init && ashlar add button form-field text-input textarea alert dialog` works end-to-end on a fresh Next.js app, Vite app, and plain HTML page without bespoke setup.
- [ ] `ashlar search` and `ashlar view button` show code, dependencies, evidence status, platform features, and policy mappings before install.
- [ ] `ashlar init` writes AGENTS.md and DESIGN.md; DESIGN.md passes `@google/design.md lint` where the alpha tool is available.
- [ ] `ashlar audit --policy federal` flags missing page title, missing meta description, missing banner, and incomplete identifier required links.
- [ ] `ashlar update` cleanly merges changes across 30 instrumented test scenarios (clean replace, codemod-applied, 3-way clean merge, 3-way conflict). Conflict rate under 15% (relaxed for v0.0; target <10% for v0.1).
- [ ] `ashlar audit` correctly flags icon-only buttons without `aria-label`, dialogs without title, and three other anti-patterns per component, across TSX and Twig sources.
- [ ] `ashlar verify` correctly identifies a tampered capsule (manual test by editing a file's content after install).
- [ ] All 14 capsules emit extended CEM that validates against the JSON Schema.
- [ ] All 14 capsules pass automated axe checks with zero violations.
- [ ] All 14 capsules have keyboard interaction tests (Playwright).
- [ ] At least 5 capsules reach `manual-tested` status (NVDA + VoiceOver runs).
- [ ] Evidence packets for Button, FormField, Dialog, Banner, and Identifier include WCAG and ICT Baseline mappings where applicable.
- [ ] Bundle size for the 13 L0 capsules combined is under 16KB gzipped (CSS only).
- [ ] Bundle size for the 1 L1 capsule (ComboBox) is under 13KB gzipped (Lit + Zag + machine + shell).
- [ ] Three demo apps render the same components and pass accessibility audits.

## Capacity Assumption

For v0.0 to advance through its gates without quality cuts:

- 2 senior frontend engineers or equivalent agent-assisted implementation capacity.
- 0.5 designer (theme, tokens, demo app design).
- 0.25 accessibility consultant (manual screen-reader testing for 5 capsules).
- ~$15k for a11y consultant time.

If capacity differs, cut scope before lowering quality gates.

## Risks specific to v0.0

- **Three-way merge UX is harder than expected.** Mitigation: build the merge protocol first; test on 30 real-world drift scenarios before locking the design.
- **Lit custom-element + React 19 + Next.js SSR has hydration friction.** Mitigation: pick this combination as the v0.0 proving ground; if it fails, address before scaling to other components.
- **Firefox ARIA reflection gaps in form-associated custom elements.** Mitigation: ComboBox in v0.0 uses host-attribute ARIA fallback; document the discipline.
- **DTCG 2025.10 / Terrazzo issues.** Mitigation: Style Dictionary v4 as fallback; implement token compilation in a swappable layer.
- **Scope exceeds available capacity.** Mitigation: explicit out-of-scope list; defer ruthlessly. If forced to choose between scope and quality, choose quality and ship what works.

## What v0.0 explicitly does not promise

- Framework parity beyond React.
- Production-ready stability (it is alpha; consumers must opt in).
- USWDS feature parity (we ship 13 capsules, not 47).
- Compliance certification (component-level evidence ≠ app-level compliance).

## What v0.0 unlocks

If v0.0 succeeds:

- We know the architecture works at small scale and can scale.
- We have signed-registry, lockfile, drift, and CEM infrastructure to build on.
- We have a real cross-stack demo we can show partners.
- We can start the public alpha (v0.1) with confidence.

If v0.0 partly fails — say, the merge UX is unclean or the bundle is too big — we redesign before broader investment. **This phase exists to falsify the architecture, not just to ship features.**

# v0.0 ŌĆö Foundation

The smallest viable substrate that proves Ashlar's architecture. Internal alpha plus two or three external design partners. **No public release.**

v0.0 is staged as **six independently shippable slices**, not one monolithic release. Each slice exits with its own success criteria. The substrate (validator, signed registry, drift management, AI contracts, token pipeline) ships before the components scale. This is a deliberate inversion of the usual design-system order: **infrastructure first, components second**.

If any slice cannot meet its exit criteria after one redesign, the architecture needs a deeper revision before further investment.

## Goal in one sentence

Demonstrate that a federal contractor can run `ashlar audit` against an existing project, install signed component capsules with safe updates, see machine-readable accessibility evidence, and have an AI coding agent validate generated code against an extended Custom Elements Manifest ŌĆö without depending on a single framework, a single template language, or a single component library.

## Slice 1 ŌĆö Standards & Evidence Ō£ō (shipped)

**Status**: shipped on `codex/standards-evidence-slice`.
**Spec**: `docs/superpowers/specs/2026-04-29-v0-0-standards-evidence-slice-spec.md`.

Delivers: registry index, registry helper library, federal HTML policy audit (parser-backed, seven page-shell rules), SARIF output, `search`/`view`/`design sync` commands, evidence packet with ICT Baseline planned mappings, Button capsule metadata enrichment.

## Slice 2 ŌĆö Validator Wedge Ō£ō (shipped code-only)

**Goal**: turn extended CEM `_ashlar.antiPatterns` data into executable rules. Deliver the *adoption wedge*: federal contractors can run `ashlar audit` in CI against existing projects with no Ashlar components installed.

**Spec**: `docs/superpowers/specs/2026-04-29-v0-0-validator-wedge-spec.md`.

**Status**: shipped in code. The npm distribution criterion remains tied to Slice 4 release hardening, so the validator wedge is demonstrable from a local checkout before it is externally consumable as `npx @blen/ashlar`.

**Scope**:
- Bundle ast-grep as an optionalDependency per platform (npm pattern).
- CEM-to-ast-grep YAML compiler in `@blen/ashlar-cli` (reads `_ashlar.antiPatterns`, emits ast-grep rules).
- Audit command supports component anti-pattern checks across HTML and TSX (the languages with first-party ast-grep coverage).
- `--explain` output includes WCAG criterion citations from the CEM and federal-standards links from policy packs.
- SARIF output carries line and column from parse5 source offsets and ast-grep ranges.
- CI workflow runs `evidence --check`, `audit --policy all` against the compliant plain HTML example, and `audit --policy federal` against a non-Ashlar legacy federal fixture so the adoption-wedge SARIF is non-empty.
- Validator returns `language-unsupported` cleanly when asked to scan a target language whose grammar is not present.
- Standalone `audit` distribution path documented (npm + planned Rust binary follow-on).

**Out of scope**:
- Vue, Svelte, Astro support (custom-language registration documented but not bundled).
- Twig, Jinja, Nunjucks (no maintained tree-sitter grammars).
- Codemod runner (rules can be authored in the validator slice; codemod application is handled by Slice 3).
- New components.

**Exit criteria**:
- `ashlar audit` flags Button anti-pattern (`icon-only-needs-label`) across an HTML and a TSX fixture.
- `--explain` shows WCAG 4.1.2 citation and a federal-standards link where applicable.
- SARIF includes `region.startLine` / `startColumn` for every finding.
- CI runs federal audit on `examples/legacy-federal-project/index.html` and surfaces non-empty SARIF while the compliant reference page stays clean.
- `audit --language twig` returns a `language-unsupported` finding rather than crashing or silently passing.
- A federal contractor can `npx @blen/ashlar@<tag> audit` in CI without any other Ashlar artifact.

## Slice 3 ŌĆö Drift Management

**Goal**: prove the shadcn-killer differentiator. Source ownership with safe updates.

**Status**: prototype shipped for the v0.0 substrate. `ashlar update` can clean-replace, clean-merge with `git merge-file --diff3`, write conflict markers, finalize via `--resolved`, refresh lockfile hashes, apply integrity-covered capsule codemods, require approval for accessibility-critical updates, and emit a survival report. `update-survival.test.ts` covers 11 throwaway-consumer scenarios plus a real `examples/vite` consumer update/build survival check. This remains a prototype until broader capsule formats and real release traffic exercise it.

**Scope**:
- `ashlar update` command with per-file three-way merge using `git merge-file --diff3`.
- Codemod runner (ast-grep apply) for breaking changes.
- Lockfile `current_hash` refreshed on every CLI invocation (not seeded equal to `original_hash`).
- Accessibility-critical force-confirmation when a touched file has `_ashlar.criticalForA11y: true`.
- Conflict resolution UX with `<<<<<<<` markers and `update --resolved <component>` finalization.
- Instrumented test scenarios: at least 10 across {clean replace, codemod-applied, 3-way clean, 3-way conflict, semantic-edit-conflict-known-limit}.
- Documented limits of textual merge (CSS variable rename, cascade-layer reorder, JSON formatting, TS state-machine refactor) with explicit failure-mode notes.

**Exit criteria**:
- Conflict rate under 25% across the test scenarios (relaxed for prototype; v0.1 target 15%, v1.0 target 10%).
- A semantic-edit failure mode (e.g. CSS variable rename) is explicitly demonstrated and documented as a known limit, with a codemod escape hatch suggested.
- Accessibility-critical confirmation prompts on a clean merge that touches `button.css`.
- One real consumer (the Vite example) survives an update through a simulated breaking change.

## Slice 4 ŌĆö Supply-chain Hardening (parallel with Slice 5)

**Goal**: real signatures, real provenance, real verification. Procurement-grade.

**Status**: started. Registry capsules now include signed `*.capsule.json` files, `registry/index.json` pins per-version capsule hashes, `registry/trust-root.json` publishes a local Ed25519 verification key plus Sigstore identity/issuer policy and `bundleVerification: "cosign"`, schema validation covers those fields, `ashlar add` / `ashlar update` verify the manifest contents, index-pinned hash, local signature, any declared Sigstore bundle metadata, and trust-root-required `cosign verify-blob` before copying or merging source, `ashlar verify` rejects lockfile capsule-hash drift, stale registry manifests, bad signatures, invalid declared Sigstore material, and failed cosign verification, `ashlar registry mirror` creates a verified local directory mirror, `ashlar release sign-capsules` can generate capsule-local cosign bundles and update manifests with verified Sigstore metadata, `ashlar release public-trust-verify` can require all signed capsule bundles to pass the consumer Sigstore verifier and emit review-record JSON, `ashlar release sbom` / `attest` emit release-review artifacts, `ashlar release trust-bundle` ties the local registry trust root plus verified capsule hashes to the SBOM and attestation through `release-trust-bundle.schema.json`, `ashlar release provenance-verify-public` can verify npm's public provenance attestations after a real publish and emit review-record JSON, `.github/workflows/publish.yml` can upload `ashlar-npm-provenance.json`, `.github/workflows/sigstore.yml` can keylessly sign capsule manifests plus release-review artifacts with cosign bundles and upload `ashlar-public-trust.json`, and `docs/security/supply-chain-incident-playbook.md` is drafted and gated by `ashlar release readiness`. Real published capsule Sigstore bundles, public npm provenance, public Sigstore/TUF trust bundles, and external security review remain open.

**Scope**:
- Capsule manifest written as `*.capsule.json` per capsule directory, not computed dynamically at install. **Started: signed local manifests are present and verified by `add` / `update` / `verify`.**
- `capsule_hash` computed registry-side, embedded in `registry/index.json`, and verified on `add` / `update` against locally-recomputed hash. **Started: implemented for local registry installs with Ed25519 manifest signatures; declared Sigstore bundle metadata and trust-root-required cosign verification are wired when bundle metadata is present.**
- Sigstore signing in CI workflow using cosign sign-blob with keyless OIDC identity tied to the GitHub Actions workflow. **Started: `release sign-capsules` generates capsule bundles, `release public-trust-verify` verifies the signed registry artifact and emits review-record JSON, and `.github/workflows/sigstore.yml` runs both before the trust bundle; public bundle publication remains open.**
- `ashlar verify` validates Sigstore signatures against an embedded chain plus pinned trust roots. **Started: local Ed25519 trust-root validation is wired, declared capsule Sigstore bundle metadata is checked against trust-root identity/issuer policy, and `sigstore.bundleVerification: "cosign"` runs `cosign verify-blob`; real public capsule bundles remain open.**
- Air-gapped operation: `ashlar registry mirror --registry ./registry --output ./mirror` produces a verified local directory mirror with capsule contents plus the local `trust-root.json`. **Started: directory mirror works with local signatures, declared Sigstore metadata checks, and trust-root-required cosign verification; compressed artifacts, Rekor/TUF trust material, and revocation policy remain open.**
- npm provenance attestations on every CLI release (npm trusted publishing). **Started: package metadata, CI prerequisites, tokenless `publish.yml`, post-publish `ashlar release provenance-verify-public`, and `ashlar-npm-provenance.json` upload are gated; npmjs.com trusted publisher configuration and a real published attestation remain open.**
- Supply-chain incident playbook documented: what happens when the Ashlar npm package is hijacked, how consumers detect it, the rotation/revocation process, the `verify` failure mode. **Started: playbook exists and release readiness checks required sections; external security review remains open.**
- SBOM published with each release. **Started: local SPDX SBOM and hash attestation are generated and verified in CI; signed/public release SBOM publication remains open.**

**Exit criteria**:
- `ashlar verify` rejects a tampered capsule. **Started: installed-file tampering, stale registry manifests, registry-index capsule-hash mismatch, local signature tampering, invalid declared Sigstore metadata, and failed trust-root-required cosign verification are covered; real public bundle publication remains open.**
- Sigstore signature is recorded in lockfile and re-validated by `verify`.
- Air-gapped verification works with a saved trusted root and no network access. **Started: filesystem registry mirrors verify with local Ed25519 trust roots, and release trust bundles tie local trust material to SBOM artifacts; public Sigstore/TUF validation remains open.**
- npm provenance attestation visible on the published CLI package.
- Incident playbook is reviewed by at least one external security maintainer (R3 cross-org check).

## Slice 5 ŌĆö AI Contracts (parallel with Slice 4)

**Goal**: the AI claim grounded in working code. Read-only, threat-modeled, no write tools.

**Status**: started. `ashlar mcp` now starts a local stdio server with read-only `search_components`, `suggest_for_task`, `get_component`, `get_evidence`, `list_tokens`, `get_token`, and `validate_usage` tools. `search_components` shares the CLI's ranked metadata discovery engine across component text, policy mappings, platform features, tokens, evidence status, family, and stability. `suggest_for_task` returns deterministic metadata-backed capsule recommendations and an install command without modifying files. `validate_usage` shares the CLI audit runner and is covered by an in-memory MCP integration test. Token tools expose DTCG paths plus generated CSS variable names. AGENTS.md is lockfile-aware and fans out to Claude, Cursor, Windsurf, Continue, and Copilot instruction files. DESIGN.md now refreshes from config, default stock theme tokens, lockfile, registry evidence, and capsule metadata. `ashlar ai-eval` provides a schema-backed saved-output eval harness, and ADR 0012 threat-models the v0.0 local read-only MCP/eval surface. Hosted transport, write tools, embedding search, and a real model-output corpus remain open.

**Scope**:
- MCP server: `ashlar mcp` starts a stdio server with read-only tools (`search_components`, `suggest_for_task`, `get_component`, `get_evidence`, `list_tokens`, `get_token`, `validate_usage`).
- Policy-aware component discovery in `ashlar search` and MCP `search_components`, with ranked reasons and install commands for coding agents.
- Deterministic task-to-capsule suggestions in `ashlar suggest` and MCP `suggest_for_task`, with no file writes.
- `validate_usage` runs the same policy runner as the CLI against a file path or glob and returns structured findings.
- AGENTS.md template references the actual installed-components list from the lockfile (not a hardcoded "Button + Link").
- Symlink fanout from AGENTS.md to `CLAUDE.md`, `.cursor/rules/ashlar.mdc`, `.windsurfrules`, `.continuerules`, `.github/copilot-instructions.md`. Copy fallback when symlinks are unavailable.
- Extended CEM `_ashlar` JSON Schema published in `@blen/ashlar-schemas` and validated in CI.
- DESIGN.md generated dynamically from config, theme tokens, lockfile, registry evidence, and capsule metadata.
- MCP threat model ADR before the server flips on by default.
- Eval harness skeleton that can ground a coding-agent prompt in CEM and check generated output against `_ashlar.antiPatterns`. **Implemented: `ashlar ai-eval` validates saved generated files and emits grounding metadata.**

**Out of scope**:
- Write tools (`add_component`, `update_component`, `migrate_apply`) ŌĆö gated behind a separate threat model ADR. v0.1 at earliest.
- Embedding-backed semantic search ŌĆö requires an embedding pipeline; metadata search and deterministic task suggestions are shipped, but model-backed planning remains planned.
- Hosted/remote MCP ŌĆö local-only in v0.0.

**Exit criteria**:
- `validate_usage` called from a sample MCP client returns the same findings as the equivalent CLI invocation.
- AGENTS.md correctly enumerates installed components after `add` / `update` runs.
- Editor-config symlinks resolve to AGENTS.md content on macOS and Linux; copy fallback documented for Windows.
- DESIGN.md changes when the chosen theme changes.
- `ai-eval` can run a deterministic generated-output suite and fail on missing expected anti-pattern findings.

## Slice 6 ŌĆö Token Pipeline

**Goal**: DTCG as the source of truth, not a downstream artifact. Tailwind v4 output. Real contrast validation.

**Status**: started. Stock themes are JSON-loaded, local project token JSON regenerates CSS, a Tailwind v4 `@theme` companion, and typed token helpers through `ashlar theme sync` / `add` / `update`, `ashlar theme validate` checks required semantic token paths and WCAG AA action contrast for light/dark modes, tests cover stock-theme pass plus deliberate low-contrast failure, and `examples/vite` builds with `@tailwindcss/vite` using Ashlar-backed utility classes. Broader contrast policy and design-tool exports remain open.

**Scope**:
- DTCG JSON files become the canonical token source; `theme.ts` becomes a JSON loader rather than a hardcoded TS structure. **Started: stock and local project theme JSON drive generated CSS, Tailwind companion output, and typed token helpers.**
- Compiler reads DTCG and emits CSS variables, Tailwind v4 `@theme` blocks, and typed TypeScript token names. **Started: CSS variables, Tailwind `@theme`, and typed TypeScript token helpers are wired.**
- Pin one canonical CSS variable namespace (decision: `--ashlar-color-action-primary-bg`, with `color`/`radius`/`text`/etc. segments matching Tailwind v4's `@theme` namespaces). Sweep the docs.
- Define `radius.control` (or rename Button consumption to `radius.sm` / `radius.md`).
- Contrast validation: every theme's action color pairs validate to WCAG AA against the action foreground. **Started in `ashlar theme validate`; not wired as an `audit --policy theme` policy.**
- Forced-colors fallback explicit in every theme.
- `ashlar theme sync` and `ashlar theme validate` commands. **Implemented for local CSS regeneration, required tokens, and action contrast.**

**Exit criteria**:
- Editing a DTCG-shaped JSON file regenerates CSS, Tailwind `@theme`, and typed token helpers on `ashlar theme sync`, `add`, and `update`.
- Tailwind v4 `@theme` output maps utilities to the same component CSS variables; `examples/vite` now proves consumption in a Tailwind-enabled Vite app.
- `ashlar theme validate` flags a deliberately-bad-contrast theme. `audit --policy theme` remains an optional follow-on if theme validation is folded into audit.

## v0.0 GATE

A v0.0 release is declared only when all six slices have shipped and:

- The CLI surface is `init`, `add`, `audit`, `verify`, `evidence`, `evidence collect`, `evidence apply`, `evidence prepare-stable`, `evidence review-status`, `evidence --check`, `search`, `view`, `design sync`, `update`, `mcp`, `ai-eval`, `bundle budget`, `theme sync`, `theme validate`, and `registry mirror`.
- A first public-service flow proof composes Button, Banner, Alert, Form Field, Text Input, and Error Summary capsules, with one passing fixture and one intentional failing fixture under `audit --policy all`.
- One stable-evidence component (Button) ships with axe + keyboard + at least one screen-reader transcript and full WCAG/ICT Baseline mappings. Tiered evidence: this is a *partial-stable* bar (not the full LTS three-SR matrix).
- The plain-HTML example is the federal-compliant audit target; the Vite example is a theme/workbench proof, not a federal page audit target.
- A federal contractor demo apps proof: `ashlar audit` running in CI against an existing federal-project HTML/TSX codebase produces meaningful findings without any Ashlar components installed. **Started:** `examples/legacy-federal-project/index.html` is an HTML proof fixture with no Ashlar components and four federal page-shell findings.
- Bundle target met for the Button markup primitive (under 4KB gzipped CSS and 0 B JavaScript) and checked in CI with `ashlar bundle budget button` using the capsule manifest's integrity-covered `bundleBudget` defaults.
- Documentation matches STATUS.md; no doc claims a feature that does not exist.
- One named external maintainer (different organization from the originating one) is publicly listed.

## What v0.0 explicitly does not promise

- More than one stable-evidence component (Button only; the others remain `experimental` until their slices land in v0.1).
- Interactive components with statecharts. The interactive component substrate (whether Lit + Zag or another statechart library) is a research bet, deferred to v0.1.
- Multi-framework adapters. React adapter is v0.1.
- Service patterns or application-block templates beyond Nunjucks. Defer to v0.1+.
- Section 508 application-level certification. Component-level evidence ŌēĀ app-level compliance.
- USWDS feature parity.
- Twig / Jinja / Nunjucks ast-grep validation. Returns `language-unsupported` until grammars exist.

## Capacity assumption

For v0.0 to advance through six slices without quality cuts:

- 1 senior engineer (architecture, registry, drift, supply chain).
- 1 mid-level engineer (CLI, validator wedge, AI contracts).
- 0.25 designer (theme tokens, DESIGN.md export).
- 0.25 accessibility consultant (manual keyboard and screen-reader testing for Button stable-evidence).
- ~$8k for a11y consultant time on Button alone (1-2 SR transcripts + content review).

If the team is smaller, slices 4 and 5 collapse into sequential delivery rather than parallel, and slice 6 may slip to v0.1.

If capacity differs materially from this, cut **scope before quality gates**. The kill criterion (R6 in [risks-and-mitigations.md](./risks-and-mitigations.md)) applies.

## Risks specific to v0.0

- **Slice 2 ast-grep distribution complexity.** Bundling per-platform binaries via npm optionalDependencies is well-understood but adds CI complexity. Fallback: postinstall script with explicit fail-loud message if no binary matches the platform.
- **Slice 3 textual merge fails silently on semantic edits.** Mitigation: explicitly document the failure modes; require codemod for any breaking change that affects token references, cascade-layer order, or state-machine identity.
- **Slice 4 Sigstore air-gapped operation.** Bundling Rekor inclusion proofs and TUF trust roots for offline verification is non-trivial. Mitigation: spec the BYO-TUF flow before implementation.
- **Slice 5 MCP write-tool pressure.** External users will request `add_component` and `update_component` in MCP. Mitigation: explicit "no write tools until threat model ADR" gate; deny early to avoid scope creep.
- **Slice 6 token pipeline inversion is more invasive than it looks.** Stock themes are JSON-driven, but the consumer project still treats copied token JSON as a downstream artifact; flipping to DTCG-as-source touches `init`, `styles`, the example apps, and the schema. Plan for two iterations.
- **Single-author commit history.** R3's three-org kill criterion is currently being met. Slice 1 already shipped under single-author; Slice 2 must publicly recruit at least one external reviewer.

## What v0.0 unlocks

If v0.0 succeeds:
- Federal contractors have an adoption path (validator-only) that requires no buy-in.
- The signed-registry + lockfile + drift substrate is real and demonstrable to procurement.
- The MCP read-only server gives a concrete answer to "what does AI-native mean."
- The architecture is falsified at small scale; v0.1 scales components on top of a substrate that works.

If v0.0 partly fails ŌĆö say, slice 3 conflict rate is unmanageable, or slice 5 MCP scope creeps ŌĆö the substrate gets fixed before adding components. **This phase exists to falsify the architecture, not to ship features.**

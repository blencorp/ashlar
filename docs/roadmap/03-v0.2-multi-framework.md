# v0.2 — Multi-framework, patterns, codemods

The expansion phase. Vue / Svelte / Solid adapters land. The service pattern set grows from 1 to 6. Date Picker and Data Grid finally land. Codemods become the standard upgrade mechanism. `@blen/ashlar-tailwind` companion ships.

## Goal in one sentence

Make Ashlar a credible USWDS alternative for federal teams across React, Vue, Svelte, Solid, Drupal, and plain HTML — with the service pattern coverage to support real service flows end-to-end.

## Deliverables

### Framework adapters

- `@blen/ashlar-vue` — Vue 3.5+ with composition API and slots. Auto-generated from extended CEM.
- `@blen/ashlar-svelte` — Svelte 5 with runes. Auto-generated.
- `@blen/ashlar-solid` — Solid Signals integration. Auto-generated.
- Adapter generation pipeline (`@lit/labs/gen-wrapper-*`-style) productionized.
- Per-framework SSR documentation with worked examples (Next.js, Nuxt, SvelteKit, SolidStart, Astro).

### service patterns

In addition to the address-form pattern from v0.1, ship five more:

- **Eligibility check** — branching wizard, save-and-resume.
- **Document upload** — chunked, resumable, progress, drag-drop.
- **Account creation shell** — email/SSO/Login.gov shell (no provider binding).
- **Feedback widget** — sticky/inline, plain-language prompts.
- **Application review and submit** — multi-step wizard with cross-step validation, accessible error summary, **event-sourced** so it can be audited.

The application-review-and-submit pattern is the first event-sourced pattern in Ashlar; it lays groundwork for v0.3+ research-frontier work.

### Components

Add complex components that v0.1 deferred:

- **Date Picker (range, restricted dates, fiscal calendars)** — Zag machine; native `<input type="date">` fallback for single dates.
- **Data Table** — sort, filter, virtualize, column resize, sticky header. Zag-based.
- **File Upload** — chunked, resumable, drag-drop with previews.
- **Multi-select / Tag input**.
- **Tree** (with treegrid variant for accessibility).
- **Identifier** (gov shell element, full implementation including agency-theme integration).
- **Language selector**.

Target v0.2 component set: **~32**.

### Tailwind companion

Ship `@blen/ashlar-tailwind`:

- Tailwind v4 preset that imports `@theme` from Ashlar tokens.
- Cascade-layer guidance: utility classes coexist with Ashlar components without specificity wars.
- Optional Tailwind-source variants of markup primitives for shadcn-style consumers (separate package; not the canonical implementation).

### Templates

- Add Liquid (Shopify, Jekyll) and Razor (.NET) to the supported template languages, if there is partner demand.
- Template generation pipeline productionized with test parity gates.

### Codemods

- Codemods ship for every breaking change since v0.0.
- Migration codemods from `react-uswds` to `@blen/ashlar-react` (skeleton in v0.1, full coverage in v0.2).
- ast-grep YAML rule patterns documented for community-contributed codemods.

### Multiple agency themes

- Default theme (the v0.0 baseline, refined).
- High-contrast theme (built on `forced-colors` and explicit System Colors).
- Two example agency themes (anonymized — "Example Department of Civic Services" and "Example State Government") demonstrating override patterns and contrast validation.
- `ashlar theme validate` enforces contrast pairs.
- Theme migration codemods from USWDS Sass settings to Ashlar DTCG tokens (best-effort).

### Documentation

- Per-framework guides (React, Vue, Svelte, Solid, Astro, Drupal, Rails).
- Migration guides (from USWDS, from `react-uswds`, from shadcn).
- service pattern docs with content guidance and user-research notes per pattern.
- Recipes book (assembly recipes for common federal flows).

### Accessibility evidence

- All ~32 components reach at least `manual-tested`.
- At least 24 reach `stable-evidence`.
- Independent third-party accessibility audit of the 24 stable components.
- Public evidence dashboard showing status per component.

### Demo apps

- Real-world demo: a fictional agency benefit application, end-to-end, using Ashlar patterns. Shipped as a public reference implementation in two flavors:
  - Next.js + `@blen/ashlar-react`.
  - Drupal + application-block Twig templates.

## Success criteria

- [ ] All v0.0 and v0.1 success criteria still pass.
- [ ] All 4 framework adapters (React, Vue, Svelte, Solid) pass identical behavior tests against the same Zag machine for at least 5 representative components.
- [ ] Date Picker, Data Table, File Upload pass NVDA + VoiceOver + JAWS manual tests.
- [ ] At least 6 service patterns shipped and used in at least one real partner project each.
- [ ] Independent third-party accessibility audit completed; findings addressed.
- [ ] Two real federal-service-style demo apps in production-quality state.
- [ ] At least 3 contributing organizations (avoid bus-factor=1).
- [ ] Public adoption: 20+ teams running Ashlar in non-toy projects.
- [ ] Bundle math holds at v0.2 scale.

## Out of scope for v0.2

- **AI Assistant Panel.** v1.0+.
- **Effect-system-typed accessibility.** v0.3+ research track.
- **Full event-sourcing for all patterns.** Only the application-review pattern is event-sourced in v0.2.
- **CRDT-friendly patterns.** v0.3+.
- **Resumability runtime.** v0.3+ research track.
- **Native mobile.** Out indefinitely unless demand emerges.

## Risks specific to v0.2

- **Adapter generation tooling proves brittle.** Mitigation: make the generator pipeline one of the first v0.2 workstreams; treat it as a v0.0-level architectural commitment.
- **Date Picker / Data Table accessibility is hard.** Mitigation: lean on Zag's existing machines; do not reinvent. Manual SR testing budget is the limiting factor; plan accordingly.
- **Pattern scope creep.** Mitigation: 6 patterns is the explicit cap. Pattern proposals beyond that go to v0.3 backlog.
- **Multi-framework drift.** Mitigation: behavior tests run identical scenarios against React/Vue/Svelte/Solid adapters; failure to match is a release blocker.

## What v0.2 unlocks

- Ashlar becomes a credible USWDS alternative across the dominant federal-IT framework stacks.
- service pattern infrastructure scales: community can contribute new patterns following the established model.
- Event-sourcing pattern lays foundation for the v0.3 research frontier.
- Independent accessibility audit creates credibility for procurement conversations.

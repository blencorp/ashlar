# STATUS

The honest, live index of what Ashlar implements, what is experimental, and what is planned. Every claim in `README.md`, `docs/strategy.md`, `docs/philosophy.md`, and the architecture docs should be consistent with this file. When a doc and STATUS disagree, STATUS is canonical.

For the current prompt-to-artifact gap map against the broader "gov shadcn" objective, see [docs/roadmap/objective-completion-audit.md](docs/roadmap/objective-completion-audit.md).

Registry schemas and generated metadata still preserve an internal compatibility `layer` field, and CSS still uses native cascade `@layer`. Public surfaces should stay on product-family language.

## Implemented today

These are wired up, tested, and run as documented.

- `ashlar init|create [-c|--cwd <path>] [--yes]` — writes `ashlar.config.json`, an empty `ashlar-lock.json`, three default agency themes (Default, VA, USDA) as DTCG-shaped JSON files plus generated `theme.css`, `tailwind-theme.css`, and `tokens.ts` files, an `AGENTS.md` template with editor instruction fanout, a generated `DESIGN.md`, and a generated `src/ashlar/ashlar.css` entrypoint. `create` and `--yes` are shadcn-v4-compatible ergonomics; Ashlar does not yet scaffold framework templates.
- `ashlar status|info [-c|--cwd <path>] [--registry <path>] [--json]` — read-only adoption status for a project or checkout. It reports whether Ashlar is initialized, which capsules are installed, whether the registry is readable, current markup primitive and stable-evidence coverage, external-review proof state, and concrete next commands. It is a DX/status surface only; strict replacement-grade claims still require `ashlar release readiness`.
- `ashlar add [components...] [--all] [-c|--cwd <path>] [--yes] [--view|--dry-run|--diff]` — resolves components from the local registry, verifies the registry-authored `*.capsule.json` manifests against `registry/index.json` capsule hashes, and can preview verified capsule metadata, planned file writes, or install diffs without changing files. `--all` selects every available capsule in the configured registry. Without preview flags, it copies the manifest-listed source files into the consumer project, computes per-file SHA-256, writes the lockfile entry, and regenerates the consumer's bundled CSS and indexes.
- `ashlar audit --policy federal [files...]` — parses HTML with parse5, runs seven federal page-shell rules (page title presence and length, meta description presence and length, federal banner near the top of body, identifier presence, identifier required-link coverage). SARIF carries line/column from parse5 source offsets. Rule entries include `fullDescription`, `help`, and `properties.tags`. **Standalone-friendly**: works in a fresh project without `ashlar.config.json` or `ashlar-lock.json`; `examples/legacy-federal-project/` proves useful findings on a non-Ashlar legacy page.
- `ashlar audit --policy components [files...]` — runs CEM `_ashlar.antiPatterns` rules via [ast-grep](https://ast-grep.github.io/) over HTML/TSX/JSX/CSS files. Returns `language-unsupported` cleanly for Twig/Jinja/Nunjucks rather than silent pass. Button capsule's `icon-only-needs-label` rule is wired and detects icon-only buttons missing `aria-label` in real markup.
- `ashlar audit --policy all [files...]` — runs federal and component policies together.
- `ashlar verify` — re-hashes installed files and compares against `original_hash` in the lockfile. It also compares each lockfile `capsule_hash` against the registry index and verifies registry capsule manifests plus local Ed25519 signatures when a registry trust root is available. If a capsule declares Sigstore bundle metadata, the same registry read verifies the bundle path, bundle hash, signed payload hash, certificate identity, and OIDC issuer against the registry trust policy; when `registry/trust-root.json` sets `sigstore.bundleVerification` to `cosign`, it also runs `cosign verify-blob` with the pinned identity and issuer. Reports missing files, registry hash mismatches, stale registry manifests, bad signatures, invalid declared Sigstore material, and failed cosign verification as errors; local edits and unavailable registry provenance as warnings.
- `ashlar update [components...] [--yes|--prompt] [--survival-report <path>]` — prototype update flow for local registry installs. It verifies the latest registry `*.capsule.json` against the `registry/index.json` capsule hash before copying or merging files, applies matching integrity-covered capsule-listed JSON codemods through ast-grep `pattern` / fixed `rewrite` rules before merge, replaces clean files, runs `git merge-file --diff3` for local edits, writes conflict markers when needed, supports `ashlar update --resolved <component>`, and refreshes lockfile hashes. It requires explicit approval before touching accessibility-critical files: `--yes` for non-interactive approval or `--prompt` for a review prompt. The optional survival report writes structured per-component file counts, codemod counts, and conflict-rate data for update survival runs.
- `ashlar migrate uswds [files...] [--registry <path>] [--json]` — read-only migration planning for existing USWDS markup. It parses HTML with parse5 and static TSX/JSX `className` markup with ast-grep, detects common `usa-*` component classes and `<usa-banner>`, maps available replacements to signed Ashlar capsules, reports unavailable primitives as explicit gaps, includes source locations, and emits next commands. It does not rewrite files or claim parity.
- `ashlar registry mirror --registry <path> --output <path> [--force]` — verifies the source registry index, each listed capsule manifest, each index-pinned capsule hash, local Ed25519 signatures, any declared capsule Sigstore bundle metadata, and trust-root-required `cosign verify-blob` before copying the registry directory for offline use. Refuses to write the mirror when verification fails.
- `ashlar mcp` — starts a local read-only MCP stdio server with `search_components`, `suggest_for_task`, `get_component`, `get_evidence`, `list_tokens`, `get_token`, and `validate_usage`. `search_components` uses the same policy/feature/token/evidence-aware discovery engine as the CLI, and `suggest_for_task` returns deterministic metadata-backed capsule suggestions plus an install command without modifying files. `validate_usage` shares the CLI audit runner and accepts a file path or glob. Token tools expose stock agency-theme token paths, CSS variables, raw values, resolved alias values, reviewed provenance metadata, source retrieval dates, and token-path source hints. No write tools are exposed.
- `ashlar ai-eval --suite <path> [--registry <path>] [--json]` — runs a schema-backed AI eval suite against saved generated output files. Each case records the prompt, grounding components, expected findings, and actual audit findings from the same runner used by `audit` and MCP `validate_usage`. This is deterministic validation of generated artifacts, not a live model benchmark.
- `ashlar bundle budget [components...] [--max-css-gzip <bytes>] [--max-js-gzip <bytes>] [--registry <path>] [--json]` — verifies capsule manifests, reads the manifest-listed CSS and JavaScript runtime files, computes raw and gzipped bytes, and fails when combined CSS or JavaScript gzip size exceeds the configured budget. If CLI flags are omitted, it uses integrity-covered `bundleBudget` defaults from each capsule manifest. Current strict-readiness measurement: Button CSS is 649 B gzipped against the manifest-carried 4 KB v0.0 Button gate with 0 B JavaScript; the twelve current markup primitive capsules are 2,094 B CSS gzipped against the manifest-carried 20,992 B page target with 0 B JavaScript.
- `ashlar theme sync [--force]` — regenerates `src/ashlar/themes/theme.css`, `src/ashlar/themes/tailwind-theme.css`, `src/ashlar/themes/tokens.ts`, and the Ashlar stylesheet entrypoint from local `*.tokens.json` files. `init`, `add`, and `update` also regenerate theme outputs from local theme JSON before syncing the project.
- `ashlar theme validate [--themes-dir <path>] [--json]` — validates stock or custom agency theme token files. It checks the JSON schema, mandatory reviewed provenance metadata, public source retrieval dates, required semantic token paths, and WCAG AA contrast for primary and secondary action foreground/background pairs in light and dark modes. JSON output includes the provenance summary for each theme.
- `ashlar evidence <component>` — prints the component's evidence packet from the registry helper, with `--format text|json`.
- `ashlar evidence collect <component> --fixture <path> --output <path> [--policy components|federal|all] [--registry <path>]` — runs the Ashlar audit runner against a capsule fixture and writes a schema-backed JSON automated-evidence artifact with the component version, fixture path, policy, pass/fail status, and findings. This creates supporting evidence artifacts; it does not update signed capsule evidence packets or replace manual keyboard and screen-reader testing.
- `ashlar evidence apply <component> --artifact <path> --output <path> [--registry <path>]` — validates a generated evidence artifact, checks that the component and version match the registry, and writes a proposed evidence packet with the artifact embedded under `automatedResults`. It deliberately writes to a separate output path instead of mutating signed registry files.
- `ashlar evidence prepare-stable <component> --fixture <path> --output <dir> [--policy components|federal|all] [--registry <path>]` — writes a complete non-mutating stable-evidence reviewer bundle: automated evidence artifact, proposed evidence packet, self-contained `REVIEW.html` fixture harness, manual evidence worksheet, keyboard transcript worksheet, screen-reader transcript worksheet, prefilled `ISSUE.md` for the stable-evidence review request, `REVIEWER_CHECKLIST.md` for required observations and claim boundaries, and a README with the next validation, review, graduation, maintainer publication, and review-record commands. Generated manual artifacts remain `blocked` placeholders until a human replaces them with observed results. Button templates now include component-specific keyboard and screen-reader steps for focus visibility, Enter activation, Space activation, native button role/name announcement, and activation behavior.
- `ashlar evidence prepare-stable-all --output <dir> [--family foundations|interactive-controls|framework-adapters|service-patterns|application-blocks|all] [--policy components|federal|all] [--registry <path>]` — writes a non-mutating stable-evidence intake directory for every matching registry capsule, defaulting to foundation capsules. Legacy internal family aliases are still accepted for automation. Each capsule gets the same reviewer bundle generated by `prepare-stable`, and the batch writes `INDEX.md` with status commands and the claim boundary. This is review preparation only; it does not count as stable evidence or external proof.
- `ashlar evidence review-status <component> --review-dir <dir> [--format text|json] [--registry <path>]` — reads a prepared stable-evidence reviewer bundle, validates the proposed packet, manual review worksheet, keyboard transcript, and screen-reader transcript, detects reviewer placeholders and blocked results, simulates manual review plus graduation in memory, and exits nonzero until the bundle is actually ready. It does not write reviewed or stable evidence files.
- `ashlar evidence finalize-stable <component> --review-dir <dir> [--registry <path>]` — turns a ready prepared reviewer bundle into `*.evidence.reviewed.json` and `*.evidence.stable.json` proposal files. It first runs the same `review-status` gate and refuses to write anything while placeholders, blocked transcripts, manifest drift, or stable-evidence findings remain. This still does not publish into the signed registry; `evidence publish` remains the registry mutation step.
- `ashlar evidence manual-template <component> --output <path> [--registry <path>]` — writes a schema-backed manual-evidence starter artifact for a component, prefilled from the component's WCAG and ICT Baseline mappings. Placeholders are deliberately `known-issue` / `blocked`, including a placeholder manual-test evidence path, so the artifact cannot support `stable-evidence` until a reviewer replaces them with real keyboard and screen-reader evidence.
- `ashlar evidence transcript-template <component> --type keyboard|screen-reader --output <path> [--registry <path>]` — writes a schema-backed `manual-transcript.schema.json` starter artifact for the specific manual run. The generated transcript is valid JSON but `blocked` with reviewer placeholders, so it cannot satisfy the stable gate until a human replaces it with observed results.
- `ashlar evidence transcript-validate <component> --transcript <path> [--type keyboard|screen-reader] [--registry <path>]` — validates a manual transcript artifact against the schema, optional transcript type, registry component/version, and completed-review boundary before it is referenced from a manual evidence review. The command rejects blocked, failing, TODO, TBD, or placeholder transcript worksheets directly, so reviewer handoffs cannot mistake schema-valid templates for completed manual evidence.
- `ashlar evidence review <component> --evidence-file <path> --manual-file <path> --output <path> [--registry <path>]` — validates a schema-backed manual review artifact, checks component/version against the registry and proposed packet, merges known WCAG and ICT Baseline mapping updates, appends manual keyboard and screen-reader test records with evidence references, writes reviewer metadata and known limitations, and emits a new proposed evidence packet without mutating registry files or the input proposal.
- `ashlar evidence graduate <component> --evidence-file <path> --output <path> [--registry <path>]` — attempts to mark a reviewed proposed packet as `stable` / `stable-evidence`, runs the full evidence gate against that claimed packet, verifies local evidence references resolve to files unless they are HTTPS URLs, and only writes the graduated proposal if WCAG, ICT Baseline, automated, manual, limitations, evidence-reference, and review metadata gates pass.
- `ashlar evidence publish <component> --evidence-file <path> --signing-key <path> --key-id <id> [--registry <path>] [--output <path>] [--format json]` — publishes a graduated `stable-evidence` packet into a local registry source. It verifies the current capsule manifest/signature, checks that `--key-id` is trusted by `registry/trust-root.json`, reruns the full stable evidence gate, plans local evidence references under `evidence/`, rewrites the packet to point at those capsule-local files, rejects any existing capsule evidence target with different content, rebuilds and verifies the signed capsule manifest, then writes the copied evidence files, registry evidence file, capsule manifest, and `registry/index.json` hash/stability together. `--output` writes a JSON publication receipt that stable-evidence review records can reference.
- `ashlar evidence --check [component] [--evidence-file <path>] [--registry <path>]` — checks registry evidence gates, or checks a proposed evidence packet against the matching registry component when `--evidence-file` is provided. It passes current experimental/not-reviewed capsules, but fails any component that claims `stable` or `stable-evidence` without passing WCAG mappings, ICT Baseline mappings, automated results, at least one passing manual keyboard test with an actionable evidence reference, at least one passing manual test with a real screen-reader technology and actionable transcript/evidence reference, explicit known limitations, and review metadata. Proposed evidence file references resolve from the current project root; registry packet references resolve from the capsule directory so published transcript files stay inside the signed capsule. Local manual test transcript references must be schema-backed JSON artifacts; external references must be HTTPS URLs. `TODO`, `TBD`, and placeholder evidence strings do not satisfy the stable gate.
- `ashlar evidence --report <path> [component] [--registry <path>]` — writes a Markdown evidence report for all registry components or one named component, including evidence status, WCAG/ICT Baseline mapping counts, manual/automated evidence counts, known limitations counts, and evidence-gate findings. Stable evidence reports use the same proposed-vs-registry evidence-reference resolution as `evidence --check`.
- `ashlar search|list [query] [-q|--query <text>] [-c|--cwd <path>] [--family <family>] [--stability <status>] [--evidence <status>] [--policy <text>] [--feature <text>] [--token <text>] [-l|--limit <count>] [--json]` — policy/feature/token/evidence-aware component discovery over registry index data plus extended CEM metadata. Results include ranked reasons and an install command so humans and coding agents can find a capsule by task, platform capability, federal policy mapping, accessibility metadata, or product family rather than name only.
- `ashlar suggest <task...> [-c|--cwd <path>] [--limit <count>] [--json]` — read-only task-to-capsule suggestions for common public-service UI jobs such as benefit application flows, official-site trust markers, identifier footers, forms, textareas, simple date inputs, select controls, validation, radio choices, checkbox consent, and status messaging. It is deterministic metadata matching, not an embedding service or code-writing agent. When the task asks for controls that do not exist as signed Ashlar capsules yet, such as complex date pickers or date ranges, the report includes explicit capability gaps and native-HTML recommendations so agents do not invent unavailable capsules.
- `ashlar view|docs <component> [-c|--cwd <path>] [--json]` — prints family, stability, evidence status, platform features, policy mappings, and capsule files.
- `ashlar design sync` — regenerates `DESIGN.md` from the current config, default stock theme tokens, reviewed theme provenance, lockfile, registry evidence, capsule metadata, and validation rules.
- `ashlar release provenance-check` — checks local npm provenance readiness for the release packages. It verifies `@blen/ashlar`, `@blen/ashlar-cli`, and `@blen/ashlar-schemas` have public `publishConfig` with provenance enabled, repository metadata matching `https://github.com/blencorp/ashlar.git` plus the package directory, a CI workflow with OIDC `id-token: write`, and a tokenless manual `.github/workflows/publish.yml` workflow restricted to `refs/heads/main` that uses a GitHub-hosted runner, npmjs registry setup, disabled release-build caching, `NPM_CONFIG_PROVENANCE=true`, `pnpm release`, a post-publish `ashlar release provenance-verify-public` check, and `reports/ashlar-npm-provenance.json` from `provenance-verify-public --json`. The root `pnpm release` script runs this gate before `changeset publish`. It warns that npmjs.com trusted publisher settings are manual and still must be configured before release.
- `ashlar release provenance-verify-public [--package <name@version...>] [--npm <path>] [--registry-url <url>] [--json]` — verifies public npm provenance after a real publish. It creates a throwaway npm project with the exact release package versions, runs `npm install --ignore-scripts`, then runs `npm audit signatures --json --include-attestations` and checks the verified attestation payloads reference the Ashlar GitHub repository, `publish.yml`, and each expected package/version. `--json` writes the machine-checkable artifact expected by local release-trust review records. This is the command that can turn the public npm provenance blocker from "unverified" into evidence after package publication.
- `ashlar release readiness [--registry <path>] [--ai-eval-suite <path>] [--min-markup-primitives <count>] [--min-stable-markup-primitives <count>] [--allow-unverified-public] [--allow-local-signatures] [--json] [--report <path>] [--json-output <path>]` — aggregates the current replacement-grade gates into one schema-backed report: verified capsule manifests, minimum markup primitive coverage, stable-evidence markup primitive count, evidence gate, bundle budget, deterministic AI eval, local npm provenance prerequisites, incident playbook, external review process artifacts, external review proof records, local keyless Sigstore workflow shape, public npm provenance, and public Sigstore trust. `--report` writes a Markdown artifact for human review; `--json-output` writes the same schema-backed status for automation. Strict defaults intentionally fail today because stable markup primitive evidence, external review proof, public npm provenance, and public capsule Sigstore trust are not complete; the `--allow-*` flags are only for local prototype smoke checks.
- `ashlar release proof-plan [--registry <path>] [--stable-component <name>] [--output <path>] [--json]` — maps the current strict readiness blockers to the three external-proof tracks: Button stable evidence, public release trust, and design partner validation. The report links the tracking issues, lists required artifacts, and prints exact next commands for `evidence`, `provenance-verify-public`, `public-trust-verify`, and guarded `review-record` creation. It is action planning only; it does not count as proof.
- `ashlar release review-pack --output <dir> [--registry <path>] [--stable-component <name>]` — writes one local reviewer intake directory containing strict readiness JSON/Markdown, project status, markup primitive stable-evidence review bundles, the target component review-status JSON, evidence report, AI eval report, bundle budget report, SBOM, SBOM attestation, trust bundle, release-trust checklist, proof action plan, and design-partner checklist. It intentionally does not create completed `docs/reviews/*.md` records and does not exit nonzero just because strict readiness is still failing; the pack is review preparation, not proof.
- `ashlar release sign-capsules [components...] [--registry <path>] [--cosign <path>] [--certificate-identity <identity>] [--certificate-oidc-issuer <issuer>]` — signs registry capsule manifest payloads with `cosign sign-blob --yes --bundle`, writes capsule-local `*.sigstore.json` bundles, records bundle hash, signed payload hash, certificate identity, and OIDC issuer in each `*.capsule.json`, and verifies the result with the same consumer-path Sigstore verifier before mutating the manifest. A failed signing or verification run rolls the manifest and bundle back. This requires a trust root with `sigstore.bundleVerification: "cosign"` and real cosign credentials from CI for public release.
- `ashlar release public-trust-verify [components...] [--registry <path>] [--cosign <path>] [--json]` — verifies a signed registry artifact for public capsule Sigstore trust. It requires every selected capsule to include Sigstore bundle metadata, requires `registry/trust-root.json` to set `sigstore.bundleVerification: "cosign"`, then reruns the same manifest, index-hash, bundle-hash, signed-payload-hash, certificate identity, OIDC issuer, and `cosign verify-blob` checks that consumers use. `--json` writes the machine-checkable artifact expected by local release-trust review records. Current checked-in capsules intentionally fail this command until the Sigstore workflow has produced real bundle files.
- `.github/workflows/sigstore.yml` — manual GitHub-hosted workflow for keyless Sigstore signing of capsule manifests and release-review artifacts. It runs `release sign-capsules`, verifies the signed registry with `release public-trust-verify`, writes `reports/ashlar-public-trust.json` from `release public-trust-verify --json`, regenerates the release SBOM, SBOM attestation, and release trust bundle from that signed registry state; signs each JSON artifact with `cosign sign-blob --yes --bundle`; verifies each generated bundle with certificate identity `https://github.com/blencorp/ashlar/.github/workflows/sigstore.yml@refs/heads/main` and issuer `https://token.actions.githubusercontent.com`; and uploads the signed registry material plus release artifacts. `ashlar release readiness` checks the local workflow shape. This is not yet public Sigstore trust for capsules because the workflow has not produced and published real capsule bundles yet; the capsule verifier validates declared Sigstore bundle metadata and runs `cosign verify-blob` when the trust root requires it.
- `.github/workflows/github-packages.yml` — manual main-only GitHub-hosted workflow for authenticated BLEN-scope GitHub Packages mirrors and canaries. It requires the `publish-github-packages` confirmation, grants `packages: write`, configures `actions/setup-node` for `https://npm.pkg.github.com` with scope `@blen`, writes `@blen:registry=https://npm.pkg.github.com` plus `BLEN_GITHUB_PACKAGES_TOKEN` auth into `.npmrc`, runs `pnpm check`, `pnpm build`, and `pnpm release:smoke`, then runs `pnpm release:github-packages` with npm provenance disabled for that registry. This is not the public `npx @blen/ashlar` path and does not satisfy `npm-provenance-public`.
- `docs/security/supply-chain-incident-playbook.md` — operational playbook for package, registry, signing, trust-root, and evidence compromise. It covers triage triggers, severity levels, containment, consumer detection commands, credential/key rotation, revocation and trust-root updates, public disclosure timing, consumer remediation, and post-incident review. `ashlar release readiness` now checks that the playbook exists and carries those required sections.
- `docs/roadmap/external-review-plan.md` plus `.github/ISSUE_TEMPLATE/{stable_evidence_review,release_trust_review,design_partner_review}.yml` — external review intake for the non-local proof gates: stable accessibility evidence, public release trust, and design partner validation. `ashlar release readiness` checks that the process artifacts exist, but actual external review records remain required before replacement-grade claims.
- `ashlar release design-partner-checklist --output <path> [--registry <path>] [--legacy-fixture <path>] [--service-flow-fixture <path>] [--task <text>]` — writes a Markdown design-partner reviewer checklist for the validator wedge and service-flow proof. It contains the exact audit, search, suggest, and view commands plus product, DX, and claim-boundary checks. The checklist is a review aid, not proof; strict readiness still requires a completed `docs/reviews/design-partner-*.md` record from a real reviewer.
- `docs/reviews/README.md` — documents where completed external review records belong. Strict `ashlar release readiness` fails until `docs/reviews/` contains real `stable-evidence-*.md`, `release-trust-*.md`, and `design-partner-*.md` records. `ashlar release review-record-check` validates those records directly: stable-evidence records must point at a reviewer bundle whose `evidence review-status` is ready and a local `evidence publish --output` receipt that still matches the signed registry evidence publication, release-trust records with local registry/SBOM/attestation/trust-bundle paths must pass `release verify-trust-bundle`, local release-trust npm provenance artifacts must be `release provenance-verify-public --json` output that still matches the reviewed package versions, local release-trust Sigstore verification artifacts must be `release public-trust-verify --json` output that still matches the signed registry, and design-partner records with local screen, validator-output, or checklist paths must point at existing artifacts.
- `ashlar release sbom --output <path>` — writes an SPDX 2.3 JSON release SBOM for `@blen/ashlar`, `@blen/ashlar-cli`, `@blen/ashlar-schemas`, and declared runtime dependencies resolved from `pnpm-lock.yaml`. The generated document includes package URLs, `DESCRIBES` relationships for release packages, and `DEPENDS_ON` relationships for runtime dependencies.
- `ashlar release attest --subject <path> --output <path>` and `ashlar release verify-attestation --subject <path> --attestation <path>` — write and verify an in-toto-shaped, hash-based release artifact attestation. CI uses this to detect accidental or malicious edits to the generated SBOM between generation and artifact upload. This command is unsigned tamper evidence; keyless Sigstore signing for the generated JSON artifact happens only in `sigstore.yml`.
- `ashlar release trust-bundle --registry <path> --sbom <path> --attestation <path> --output <path> [--checklist <path>]` and `ashlar release verify-trust-bundle --bundle <path> --registry <path> --sbom <path> --attestation <path>` — write and verify a schema-backed local release trust bundle containing registry index hash, trust-root hash/public keys, verified capsule hashes/signature key IDs, SBOM hash, and SBOM-attestation hash. The command refuses to build from an invalid SBOM attestation and verification catches schema errors, artifact drift, or registry drift. `--checklist` writes a Markdown release-trust reviewer checklist with npm provenance, public Sigstore, trust-bundle, artifact-attachment, and claim-boundary checks. This is an offline-review artifact; it is not npm provenance. Keyless Sigstore signing for the generated JSON artifact happens only in `sigstore.yml`.
- `@blen/ashlar`, `@blen/ashlar-cli`, and `@blen/ashlar-schemas` package manifests — include public npm publish configuration with provenance enabled and repository metadata scoped to each monorepo package directory. This supports provenance-capable publish tooling but does not by itself prove a package has been published with provenance.
- `pnpm release:smoke` — packs `@blen/ashlar`, `@blen/ashlar-cli`, and `@blen/ashlar-schemas`, verifies tarballs do not include compiled tests or compiler-internal build info, installs the scoped `@blen/ashlar` tarball into a throwaway consumer project with local overrides for the workspace packages, confirms the `ashlar` binary is present, and runs the packed CLI against a standalone federal page-shell audit fixture.
- `pnpm repo:doctor` — maintainer health command for the shared checkout. It runs public-language checking, format checking, lint/typecheck/test, build, framework visual smoke, release tarball smoke, npm and GitHub Packages readiness preflights, `git diff --check`, strict release readiness, and the proof action plan. It writes `reports/doctor/{summary.md,summary.json,release-readiness.md,release-readiness.json,proof-action-plan.md}` and exits successfully only when local checks pass and strict readiness is blocked only by the known proof gates. It does not turn those proof gates into a release pass.
- `pnpm testing:start` — manual testing session launcher. It builds once, starts the public site, docs app, Vite + Tailwind case-board example, and vanilla/React/Next.js/Svelte/Vue case-board examples on fixed localhost ports, writes `reports/testing-session/{summary.md,summary.json,checklist.md,logs/*}`, and keeps the servers alive until Ctrl-C. `pnpm testing:start --check` boots the same targets, verifies HTTP responses, writes the same reports, and exits. `pnpm testing:start --check --visual` also runs the Playwright visual smoke, writes screenshots under `reports/example-visual-smoke/`, and records those pointers in the testing summary for reviewer bug-bash sessions.
- `registry/index.json` — local source of truth for registry discovery, including per-version `capsuleHashes` that pin each capsule manifest hash.
- `registry/components/button/0.0.1/` — one capsule with `button.css`, `button.html`, `button.cem.json`, `button.evidence.json`, and `button.capsule.json`. Status: `experimental`, evidence `not-reviewed`. CEM `_ashlar.antiPatterns` are now ast-grep-native (pattern + regex + has + not + languages).
